# Como Usar o Conselho de 6 Especialistas

## Visão Geral

Este conselho é uma base de conhecimento curada de 6 dos maiores especialistas em marketing e funis de vendas. Cada especialista tem uma expertise específica e uma perspectiva única.

## Os 6 Especialistas

| Especialista | Expertise | Quando Consultar |
|--------------|-----------|------------------|
| **Russell Brunson** | Arquitetura de Funil | Estrutura geral, jornada de cliente, value ladder |
| **Dan Kennedy** | Oferta & Copy | Headlines, copywriting, ofertas persuasivas |
| **Frank Kern** | Psicologia & Comportamento | Por que clientes agem, fricção, automação |
| **Sam Ovens** | Aquisição & Qualificação | Qualificação de leads, tráfego frio, alto ticket |
| **Ryan Deiss** | LTV & Retenção | Retenção, lifecycle, segmentação, receita recorrente |
| **Perry Belcher** | Monetização Simples | Monetização rápida, bumps, upsells, afiliados |

## Como Usar Cada Tipo de Documento

### 1. Identity (Identidade)

**Uso:** Entender quem é o especialista, seu estilo, quando consultá-lo.

**Exemplo:** "Preciso estruturar meu funil. Vou consultar Russell Brunson (Arquitetura de Funil)."

**Arquivo:** `council/identity/russell_brunson.md`

**O que você encontra:**
- Papel em 1 frase
- Expertise específica
- Estilo de comunicação
- Quando consultar / quando NÃO consultar
- Princípios fundamentais
- Perguntas recorrentes
- Red flags (o que essa mente mata)

### 2. Heuristics (Heurísticas)

**Uso:** Regras práticas do especialista. "Se X, então Y, porque Z."

**Exemplo:** "Se não há oferta clara, então não há ação esperada."

**Arquivo:** `council/heuristics/oferta_copy.md`

**O que você encontra:**
- 10-20 regras "Se/Então/Porque/Exceção"
- Checklist rápido
- Métricas de referência (Ruim/OK/Bom/Excelente)

### 3. Anti-Patterns (Anti-padrões)

**Uso:** Identificar o que NÃO fazer. Erros comuns.

**Exemplo:** "Qual é o maior erro em copywriting?"

**Arquivo:** `council/anti-patterns/dan_kennedy.md`

**O que você encontra:**
- 10 anti-patterns por especialista
- Por que dá errado
- Sinais de alerta
- Como evitar
- Links para heurísticas relacionadas

### 4. Mental Models (Modelos Mentais)

**Uso:** Frameworks de pensamento. Como o especialista pensa.

**Exemplo:** "Qual é o framework de Russell para estruturar ofertas?"

**Arquivo:** `council/mental-models/russell_brunson.md`

**O que você encontra:**
- 5-10 frameworks
- Definição simples
- Como aplicar
- Perguntas de diagnóstico

### 5. Case Library (Biblioteca de Casos)

**Uso:** Exemplos reais. Validação prática.

**Exemplo:** "Qual é um exemplo real de Russell aplicando Value Ladder?"

**Arquivo:** `council/case-library/russell_brunson.md`

**O que você encontra:**
- 5-10 casos reais
- Metadados (data, tipo, resultado)
- O que fizeram, por que, resultados
- Artefatos (links para páginas/emails)

### 6. Failure Patterns (Padrões de Falha)

**Uso:** Entender onde o especialista falha. Limitações.

**Exemplo:** "Onde o framework de Russell não funciona?"

**Arquivo:** `council/failure-patterns/russell_brunson.md`

**O que você encontra:**
- Contextos onde falha
- Tradeoffs (o que ganha vs o que perde)
- Quando usar outro especialista

### 7. Playbooks (Playbooks)

**Uso:** Passo a passo operacional. Como implementar.

**Exemplo:** "Como reviso meu funil?"

**Arquivo:** `council/playbooks/funnel_review_playbook.md`

**O que você encontra:**
- Objetivo
- Duração
- Pré-requisitos
- Etapas do playbook
- Checklist
- Padrões de teste A/B
- Métricas alvo

### 8. Scorecards (Scorecards)

**Uso:** Avaliar qualidade. Rubrica de 0-10.

**Exemplo:** "Qual é a qualidade do meu funil?"

**Arquivo:** `council/scorecards/scorecard_funnel.md`

**O que você encontra:**
- Dimensões de avaliação
- Rubrica de 0-10 por dimensão
- Peso por dimensão
- Regras de decisão (EXECUTAR / AJUSTAR / MATAR)
- Plano de ação

### 9. Decisions (Decisões)

**Uso:** Template para tomar decisões consultando o conselho.

**Exemplo:** "Devo aumentar o preço ou adicionar um bump?"

**Arquivo:** `council/decisions/decision_template.md`

**O que você encontra:**
- Informações básicas
- Problema a resolver
- Opções consideradas
- Consulta a cada especialista
- Síntese do conselho
- Decisão final
- Acompanhamento

## Fluxo de Uso Típico

### Cenário 1: Revisar um Funil Existente

1. **Comece com Identity** → Entenda qual especialista consultar
2. **Vá para Scorecard** → Avalie a qualidade atual
3. **Consulte Heuristics** → Identifique o que melhorar
4. **Veja Anti-patterns** → Evite erros comuns
5. **Use Playbook** → Implemente melhorias
6. **Meça Resultados** → Valide com métricas

### Cenário 2: Criar uma Nova Oferta

1. **Comece com Identity** → Qual especialista? (Dan Kennedy para copy, Perry Belcher para monetização)
2. **Consulte Mental Models** → Qual framework usar?
3. **Veja Case Library** → Qual é um exemplo similar?
4. **Consulte Heuristics** → Quais são as regras?
5. **Use Playbook** → Como implementar?
6. **Use Scorecard** → Avalie a oferta

### Cenário 3: Tomar uma Decisão Importante

1. **Use Decision Template** → Documente o problema
2. **Consulte cada Identity** → Entenda perspectivas
3. **Consulte Heuristics** → Quais são as regras?
4. **Consulte Failure Patterns** → Onde pode falhar?
5. **Documente Síntese** → Qual é o consenso?
6. **Tome Decisão** → Baseado em conselho

## Dicas de Uso

### Dica 1: Comece com Identity

Sempre comece lendo a Identity do especialista. Entenda seu estilo, quando consultá-lo, e quando NÃO consultá-lo.

### Dica 2: Consulte Múltiplos Especialistas

Raramente um problema tem apenas uma perspectiva. Consulte 2-3 especialistas relevantes.

**Exemplo:** Para estruturar uma oferta:
- Russell Brunson (onde na ladder?)
- Dan Kennedy (como escrever?)
- Perry Belcher (como monetizar?)

### Dica 3: Use Heuristics para Diagnóstico Rápido

Se precisa de resposta rápida, vá direto para Heuristics. Se/Então regras são rápidas de entender.

### Dica 4: Use Playbooks para Implementação

Se precisa de passo a passo, use Playbooks. Eles têm checklist e métricas.

### Dica 5: Use Scorecards para Avaliação

Se precisa avaliar qualidade, use Scorecards. Eles têm rubrica clara.

### Dica 6: Consulte Failure Patterns Antes de Implementar

Antes de implementar algo, leia Failure Patterns. Entenda onde pode falhar.

### Dica 7: Documente Decisões

Use Decision Template para documentar decisões importantes. Isso cria histórico e aprendizado.
